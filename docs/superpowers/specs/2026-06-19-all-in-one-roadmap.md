# All-In-One Discord Bot — Master Build Roadmap

**Target:** Match ProBot + Scripted Elite AIO + Invite Management, **excluding Double Counter.**
**Stack (fixed):** discord.js v14 · CommonJS · JSON storage via `src/core/store.js` · slash commands via `commandModules` + `bypassModGate` · interaction routers like `src/tickets/interactions.js`.

> Generated by a 7-agent research+audit workflow (ProBot/AIO/Invite-Mgmt feature research, open-source reference hunt, full code audit). Open-source copy-sources are MIT-licensed unless noted.

---

## 0. Hard prerequisites (block everything)

### 0.1 Server "Require 2FA for moderation" → bot owner needs 2FA
If the server requires 2FA for moderation (default for Community servers), **any moderation-class action by the bot fails with error `40002` unless the Discord account that OWNS the bot app has 2FA enabled.** Affects ban/kick/mute, lockdown, anti-nuke/raid, and **ticket channel create/delete** (the most-used feature — this is likely why "tickets won't connect").
→ **Enable 2FA on the bot-owner account (Devloop7).**

### 0.2 Railway persistent storage (silent data loss)
Railway's filesystem is **ephemeral** — every redeploy wipes `data/` (all tickets, warnings, strikes, configs, panel IDs gone).
→ **Mount a Railway Volume and set `BOT_DATA_DIR` to its mount path** (e.g. `/data`). `store.js` already honors `BOT_DATA_DIR`. Add a startup log of the resolved data dir + writability.

### 0.3 Intents
Have: `MessageContent`, `GuildMembers`. Add later: `GuildInvites` (invites), `GuildVoiceStates` (temp voice), `GuildMessageReactions` + Message/Reaction partials (only if classic reaction-roles).

## 2. Cross-cutting infrastructure (build once, in Phase 1)

- **2.1 Lock-safe async store:** add `store.mutate(name, fn)` — a per-file in-process write queue so concurrent writes (strikes/xp/invites/tickets) don't lose updates. Route all hot counters through it.
- **2.2 Per-guild settings (`src/core/guildConfig.js`):** one `guild-settings.json` keyed by guildId with a DEFAULTS-merge pattern (like ticketStore) holding welcome/invites/leveling/logging/automod/reactionRoles/autoresponders config. Migrate modlog to read it (fallback to `config.js`).
- **2.3 Router convention:** each feature owns one `register(client)` listening on `InteractionCreate`, early-returning unless its customId namespace matches. Namespaces: `eb:` embed, `rr:` roles, `gw:` giveaway, `tv:` voice, `inv:` invites, `lvl:` leveling, `sc:` scripts.
- **2.4 `src/core/perms.js`:** shared staff check (ManageGuild + `guildConfig.staffRoleIds`).

## 3. Phased build order

### PHASE 1 — Urgent client needs + must-fix bugs (ship first; release gate)
**1A Embed/announcement poster:** `/embed` (title/description/color/image/thumbnail/footer/author/ping + channel), `/say` (plain text), `/embedbuilder` (interactive buttons+modals, `eb:` namespace, in-memory draft Map), `/embededit`. Staff-gated; preflight bot perms in target; always set `allowedMentions`; mirror to mod-log. Ref: Kaitorm/Slash-Command-Embed-Builder (MIT), discordjs.guide embeds.
**1B Ticket quicksetup:** `/ticket quicksetup` zero-arg wizard (auto-create category + logs + panel); setup/startup **permission preflight**; catch `40002` and reply "enable 2FA on owner account"; **persist ticket-delete timer** (deleteAt + boot sweep). Ref: ralphkb/sentinel-tickets (MIT, patterns).
**1C Bug fixes (from audit):** B1 strike decay + `/clearstrikes`; B2 shared "already actioned" guard (profanity+links); B3 kick/mute fetch uncached member; B4 hierarchy/self/bot-target guards; B5 dispatcher check `replied||deferred`; B6 anti-nuke audit-entry dedupe; B7 persist raid-lift + ticket-delete across restart (`src/core/scheduler.js`); B8 restore original verification level; B9 profanity word-boundary (no substring false positives); B10 modlog via guildConfig; store race (2.1); Railway persistence (0.2); bot-permission preflight on ready.

### PHASE 2 — Welcome/leave (+ image card via `@napi-rs/canvas` MIT, bundle a .ttf) + auto-role
Refs: Tomato6966/Discord-Welcome-Images (MIT), AKORA-Studios/DiscordWelcomeCard (MIT). Avoid canvacord (GPL+archived), discord-arts (unlicensed).

### PHASE 3 — Invite tracking (cache-and-diff on ready/guildMemberAdd; `GuildInvites` + ManageGuild). `real = joined − left − fake + bonus`; persist `joinedBy`. Ref: Androz2091/discord-invites-tracker (MIT).

### PHASE 4 — Leveling/XP (per-msg XP + cooldown, curve, level roles, rank card). Route writes via `store.mutate`. Ref: saiteja-madha/discord-js-bot (MIT, CommonJS) leveling module.

### PHASE 5 — Button/select self-roles (`rr:` namespace; no extra intents). Ref: saiteja-madha (MIT), OMJO-MOJO/Discord.js-Menu-Roles.

### PHASE 6 — Expanded automod (caps%/emoji/dupe + mention warn-tier; optional native AutoMod sync) + link bare-domain fix + mod exemption.

### PHASE 7 — Full event logging (generalize modlog; msg edit/delete, joins, roles, channels, voice; attribute via auditlog). Ref: devdeem/Logger-Bot.

### PHASE 8 — Giveaways (button entry; persist endsAt + boot re-arm via scheduler). Ref: Androz2091/discord-giveaways (MIT, JSON adapter).

### PHASE 9 — Autoresponder / custom commands. Ref: saiteja-madha (MIT).

### PHASE 10 — Temp voice (join-to-create, `GuildVoiceStates`) + stat counters (debounced renames). Ref: itzzkirito/VoiceMaster, saiteja-madha.

### PHASE 11 — Optional: Starboard + Script catalog (`/script add|list` + download buttons, built on Phase-1 embed/button infra; `scripts.json`).

## 4. Library picks (Railway/Windows clean-build)
- **Image cards:** `@napi-rs/canvas` (MIT, prebuilt binaries, zero system deps). Bundle a `.ttf` + `GlobalFonts.registerFromPath`.
- **Giveaways (optional):** `@androz2091/discord-giveaways` (MIT, pluggable JSON storage).

- **Avoid for verbatim copy:** GPL repos (discord-tickets/bot, open-ticket, Bes-js/discord-invite), canvacord (GPL+archived), discord-arts (unlicensed).

## 5. Execution contract (every phase)
Add slash module to `commandModules` (auto-registers); add `register(client)` to `index.js` `modules`; store config in `guildConfig`; route hot writes via `store.mutate`; namespace customIds; keep `node --test` green (42/42) + add core-logic tests. **Phase 1 is the release gate** — don't ship later phases until 1A/1B/1C land, the Railway volume is mounted, and owner 2FA is confirmed.
